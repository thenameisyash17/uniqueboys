DBMS Fundamentals

What is DBMS?

A Database Management System (DBMS) is software that allows users to define, create, maintain, and control access to databases. It provides an interface between the database and end users or application programs.

Advantages of DBMS

  • Data Redundancy Control: Minimizes duplicate data
  • Data Integrity: Ensures accuracy and consistency
  • Data Security: Provides access control mechanisms
  • Backup & Recovery: Automatic backup and restore capabilities
  • Data Independence: Separation of data and applications

Database Architecture

3-Tier Architecture:

  • External Level: User views
  • Conceptual Level: Database structure
  • Internal Level: Physical storage
Types of Database Models

Hierarchical Model

Tree-like structure with parent-child relationships. Used in early mainframe databases.

Network Model

Graph structure allowing many-to-many relationships. More flexible than hierarchical.

Relational Model

Data organized in tables (relations) with rows and columns. Most widely used today.

NoSQL Model

Non-tabular databases like document, key-value, graph, and column-family stores.

Entity-Relationship Model

Entity

A real-world object that can be distinctly identified. Examples: Student, Course, Employee.

Types: Strong Entity (has primary key), Weak Entity (depends on another entity)

Attributes

  • Simple: Cannot be divided (e.g., Age)
  • Composite: Can be divided (e.g., Name → First, Last)
  • Derived: Can be calculated (e.g., Age from DOB)
  • Multi-valued: Multiple values (e.g., Phone Numbers)
  • Key: Uniquely identifies entity

Relationships

Association among entities with degree and cardinality:

  • Degree: Unary, Binary, Ternary
  • Cardinality: One-to-One, One-to-Many, Many-to-Many

Data Definition Language (DDL)
CREATE TABLE Example
CREATE TABLE Students (
    student_id INT PRIMARY KEY,
    first_name VARCHAR(50) NOT NULL,
    last_name VARCHAR(50),
    email VARCHAR(100) UNIQUE,
    date_of_birth DATE,
    enrollment_date DATE DEFAULT CURRENT_DATE,
    gpa DECIMAL(3,2) CHECK (gpa >= 0 AND gpa <= 4.0),
    department_id INT,
    FOREIGN KEY (department_id) REFERENCES Departments(department_id)
);

SQL Joins & Advanced Queries

INNER JOIN

SELECT s.student_id, s.first_name, e.grade
FROM Students s
INNER JOIN Enrollments e 
ON s.student_id = e.student_id;

LEFT JOIN

SELECT d.department_name, COUNT(s.student_id)
FROM Departments d
LEFT JOIN Students s 
ON d.department_id = s.department_id
GROUP BY d.department_name;

Subquery

SELECT first_name, last_name, gpa
FROM Students
WHERE gpa > (
    SELECT AVG(gpa) 
    FROM Students
);

Database Normalization

Purpose

Normalization is the process of organizing data to reduce redundancy and improve data integrity by dividing tables and defining relationships between them.

Normal Forms

  • 1NF: Eliminate repeating groups
  • 2NF: Remove partial dependencies
  • 3NF: Remove transitive dependencies
  • BCNF: Boyce-Codd Normal Form
  • 4NF: Remove multi-valued dependencies
  • 5NF: Remove join dependencies
Normalization Example

Unnormalized Table

Student_Course (student_id, student_name, course1, course2, course3, instructor1, instructor2)

After 1NF

Student (student_id, student_name)
Enrollment (student_id, course_id, instructor_id)
Course (course_id, course_name)

After 2NF & 3NF

Student (student_id, student_name)
Course (course_id, course_name, instructor_id)
Instructor (instructor_id, instructor_name)
Enrollment (student_id, course_id, grade)

Transactions & Concurrency Control

ACID Properties

  • Atomicity: All or nothing execution
  • Consistency: Database remains consistent before and after
  • Isolation: Transactions execute independently
  • Durability: Committed changes persist

Concurrency Problems

  • Dirty Read: Reading uncommitted data
  • Lost Update: Overwriting changes
  • Non-repeatable Read: Different reads of same data
  • Phantom Read: New rows appear between reads

Concurrency Control Methods

  • Lock-Based: Two-phase locking
  • Timestamp-Based: Order by timestamp
  • Optimistic: Validate before commit
  • Multiversion: MVCC (used in PostgreSQL)
Transaction Example in SQL
BEGIN TRANSACTION;

-- Transfer $100 from account 101 to 102
UPDATE Accounts SET balance = balance - 100 
WHERE account_id = 101;

UPDATE Accounts SET balance = balance + 100 
WHERE account_id = 102;

-- Check if any account has negative balance
IF EXISTS (SELECT * FROM Accounts WHERE balance < 0)
BEGIN
    ROLLBACK TRANSACTION;
    PRINT 'Transaction failed: Negative balance';
END
ELSE
BEGIN
    COMMIT TRANSACTION;
    PRINT 'Transaction successful';
END
