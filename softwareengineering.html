Software Engineering Basics

What is Software Engineering?

It is the application of engineering principles to software development to ensure reliability, efficiency, and maintainability. Software engineering involves systematic, disciplined, and quantifiable approaches to software development.

Q&A

What are the key objectives of software engineering?
The key objectives are to improve software quality, increase productivity, enable better cost estimation, manage complexity, and facilitate team collaboration.
Differentiate between software engineering and computer science.
Computer science focuses on theory and fundamentals of computation, while software engineering applies these principles to design, develop, test, and maintain practical software systems.

Software Characteristics

  • Intangible - Cannot be touched or felt physically
  • Developed, not manufactured - No classical manufacturing process
  • Highly complex - More complex than any human-made system
  • Maintenance intensive - Requires continuous updates and fixes
  • Does not wear out - But can become obsolete
  • Flexible - Can be changed relatively easily

Q&A

Why is software maintenance intensive?
Software needs constant updates due to changing requirements, bug fixes, performance improvements, security patches, and compatibility with new hardware/software environments.

Software Types

  • System Software (OS, drivers, utilities)
  • Application Software (Word processors, spreadsheets)
  • Embedded Software (Firmware in devices)
  • Web Applications (Browser-based apps)
  • Real-time Software (Control systems, robotics)
  • Scientific Software (Simulations, modeling)

Q&A

What is the difference between system software and application software?
System software manages computer hardware and provides a platform for applications (e.g., OS). Application software performs specific tasks for users (e.g., MS Word, Photoshop).

Software Development Life Cycle (SDLC)

SDLC Phases

  • Requirement Analysis - Gathering and documenting requirements
  • Design - Creating system architecture and design
  • Implementation - Coding and unit testing
  • Testing - System integration and validation
  • Deployment - Release to production environment
  • Maintenance - Updates, fixes, and enhancements

Q&A

Why is requirement analysis the most critical phase?
Errors in requirement analysis are the most expensive to fix later. Clear requirements prevent scope creep, reduce rework, and ensure the final product meets user needs.
What is the difference between verification and validation?
Verification checks "Are we building the product right?" (process-focused). Validation checks "Are we building the right product?" (user needs-focused).

Process Models

  • Waterfall - Sequential, non-iterative
  • Incremental - Divide system into builds
  • Spiral - Risk-driven iterative model
  • Agile - Flexible, iterative, customer-focused
  • V-Model - Extension of waterfall with testing phases
  • Prototyping - Build prototypes for feedback
  • RAD - Rapid Application Development

Q&A

When to use Waterfall model?
When requirements are clear and stable, technology is understood, project is short, and no ambiguous requirements exist.
List advantages of Agile methodology.
Customer satisfaction, early delivery, flexibility to change, continuous improvement, face-to-face communication, and sustainable development pace.

Requirements Engineering

Requirement Types

  • Functional - What the system should do
  • Non-Functional - Quality attributes (performance, security)
  • User Requirements - High-level statements in natural language
  • System Requirements - Detailed specifications
  • Business Requirements - High-level organizational needs
  • Stakeholder Requirements - Needs of individuals/groups

Q&A

Differentiate between functional and non-functional requirements.
Functional requirements define system behavior (e.g., "user can login"). Non-functional requirements define quality attributes (e.g., "system responds in < 2 seconds").
What are the characteristics of good requirements?
Clear, concise, unambiguous, consistent, complete, verifiable, traceable, and feasible.

SRS & Requirements Process

Software Requirement Specification is a detailed description of system requirements. The requirements engineering process includes elicitation, analysis, specification, validation, and management.

Q&A

What is requirements traceability?
The ability to track requirements through development lifecycle, from origin to implementation and testing, ensuring all requirements are addressed.
List common requirements elicitation techniques.
Interviews, surveys, observation, prototyping, workshops, brainstorming, document analysis, and use cases.

Software Design

Design Concepts

  • Abstraction - Hide complex details, show essentials
  • Modularity - Divide system into components
  • Cohesion - Strength of relationship within a module
  • Coupling - Degree of interdependence between modules
  • Encapsulation - Bundling data with methods
  • Information Hiding - Hide implementation details
  • Separation of Concerns - Divide program into distinct sections

Q&A

What is the difference between cohesion and coupling?
Cohesion measures how strongly related functions within a module are (high cohesion is good). Coupling measures interdependence between modules (low coupling is good).
Why is information hiding important?
It reduces system complexity, allows implementation changes without affecting other modules, and improves maintainability.

Design Models & Diagrams

  • Data Flow Diagram (DFD) - Shows data flow through system
  • UML Diagrams - Standard modeling language
  • ER Diagrams - Entity-relationship modeling
  • Class Diagrams - Static structure of system
  • Sequence Diagrams - Object interactions over time
  • State Diagrams - States and transitions
  • Architectural Diagrams - High-level system structure

Q&A

What are the levels of DFD?
Level 0 (context diagram), Level 1 (major processes), Level 2 (sub-processes), and further decomposition as needed.
Name three types of UML diagrams.
Structural: Class, Component, Deployment. Behavioral: Use Case, Sequence, Activity. Interaction: Communication, Timing.

Software Testing

Testing Levels

  • Unit Testing - Testing individual components
  • Integration Testing - Testing combined modules
  • System Testing - Testing complete system
  • Acceptance Testing - User acceptance testing (UAT)
  • Regression Testing - Testing after changes
  • Performance Testing - Load, stress, scalability
  • Security Testing - Vulnerability assessment

Q&A

What is the difference between alpha and beta testing?
Alpha testing is done internally by developers/testers. Beta testing is done by actual users in real environment before final release.
What is test coverage?
A measure of the amount of testing performed, expressed as percentage of requirements, code, or functionalities tested.

Testing Techniques & Strategies

  • Black Box Testing - Testing without knowledge of internal code
  • White Box Testing - Testing with knowledge of internal code
  • Gray Box Testing - Combination of black and white box
  • Manual Testing - Human tester executes test cases
  • Automated Testing - Using scripts/tools for testing
  • Test-Driven Development (TDD) - Write tests before code
  • Mutation Testing - Modify code to check test effectiveness

Q&A

Differentiate between black box and white box testing.
Black box tests functionality without knowing internal structure. White box tests internal structure, paths, and logic with knowledge of code.
What is equivalence partitioning?
A black box technique that divides input data into equivalent partitions that are expected to be processed similarly, reducing test cases.

Project Management

Project Planning & Estimation

  • Cost Estimation - COCOMO, Function Point Analysis
  • Scheduling - Gantt charts, PERT, CPM
  • Risk Management - Identification, analysis, mitigation
  • Resource Management - Human, hardware, software
  • Quality Management - Quality planning, assurance, control
  • Configuration Management - Version control, change management
  • Communication Management - Stakeholder communication plans

Q&A

What is the difference between PERT and CPM?
PERT (Program Evaluation Review Technique) is probabilistic, used when times are uncertain. CPM (Critical Path Method) is deterministic, used when times are certain.
List types of software risks.
Technical risks (technology, complexity), management risks (planning, resources), organizational risks, and external risks (market, regulatory).

Metrics, Quality & Maintenance

  • LOC - Lines of Code
  • Function Points - Measure software functionality
  • Cyclomatic Complexity - Measure code complexity
  • Quality Assurance - Process-oriented, preventive
  • Quality Control - Product-oriented, detective
  • Software Maintenance - Corrective, adaptive, perfective, preventive
  • CMMI - Capability Maturity Model Integration
  • ISO 9001 - Quality management standard

Q&A

What are the four types of software maintenance?
Corrective (fixing bugs), Adaptive (adapting to new environment), Perfective (improving performance), Preventive (preventing future problems).
What is cyclomatic complexity?
A software metric measuring the number of linearly independent paths through a program's source code, indicating complexity and testability.

Advanced Topics

Software Architecture & Patterns

Q&A

What are advantages of microservices architecture?
Independent deployment, technology diversity, scalability, fault isolation, and easier to understand and develop.

DevOps & CI/CD

  • Continuous Integration
  • Continuous Delivery
  • Continuous Deployment
  • Infrastructure as Code
  • Containerization (Docker)
  • Orchestration (Kubernetes)
  • Monitoring & Logging
  • DevOps Culture & Practices

Q&A

Differentiate between CI and CD.
CI (Continuous Integration) is merging code changes frequently. CD can mean Continuous Delivery (ready for deployment) or Continuous Deployment (automatically deployed).

How is AI transforming software engineering?
AI assists in code generation, bug detection, test case generation, requirements analysis, project estimation, and automated refactoring.

Software Engineering Glossary

Abstraction

Hiding complex implementation details and showing only essential features.

Agile

Iterative development methodology emphasizing flexibility and customer collaboration.

Algorithm

Step-by-step procedure to solve a problem or accomplish a task.

API

Application Programming Interface - set of rules for software communication.

Beta Testing

Testing by actual users in real environment before final release.

Bug

Error, flaw or fault in a computer program causing incorrect behavior.

CI/CD

Continuous Integration/Continuous Deployment - automation of software delivery.

Cohesion

Degree to which elements within a module belong together.

Coupling

Degree of interdependence between software modules.

DevOps

Combination of development and operations practices.

DFD

Data Flow Diagram - graphical representation of data flow.

Encapsulation

Bundling data with methods that operate on that data.
