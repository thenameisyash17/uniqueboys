Explain the ACID properties of database transactions with examples

‚ñº

ACID Properties ensure reliable processing of database transactions:

1. Atomicity: Transactions are all-or-nothing operations.

  • Either all operations of the transaction complete successfully, or none do
  • Implemented using transaction logs and rollback mechanisms
  • Example: Bank transfer: Both debit and credit must succeed, or both fail

2. Consistency: Transactions preserve database constraints.

  • Database moves from one valid state to another
  • All integrity constraints (primary keys, foreign keys, checks) remain satisfied
  • Example: Account balance should never be negative after any transaction

3. Isolation: Concurrent transactions don't interfere.

  • Each transaction executes as if it's the only one running
  • Implemented using locking mechanisms or multi-version concurrency control
  • Example: Two people booking the last seat on a flight

4. Durability: Committed transactions persist.

  • Once a transaction commits, its changes survive system failures
  • Implemented using write-ahead logging and database backups
  • Example: After confirming an online purchase, the order remains even after power failure

Compare and contrast the different normal forms (1NF, 2NF, 3NF, BCNF)

‚ñº

Normalization is the process of organizing data to reduce redundancy and improve data integrity.

-- Example unnormalized table
StudentCourses(student_id, student_name, course1, course2, course3, instructor1, instructor2)

First Normal Form (1NF):

  • Eliminate repeating groups in individual tables
  • Create separate table for each set of related data
  • Identify each set of related data with a primary key
  • All attributes contain atomic values

Second Normal Form (2NF):

  • Must be in 1NF
  • Remove partial dependencies (non-prime attributes dependent on part of candidate key)
  • All non-key attributes fully functionally dependent on the primary key

Third Normal Form (3NF):

  • Must be in 2NF
  • Remove transitive dependencies (non-prime attributes dependent on other non-prime attributes)
  • Every non-prime attribute is non-transitively dependent on every candidate key

Boyce-Codd Normal Form (BCNF):

  • Stricter than 3NF
  • For every functional dependency X ‚Üí Y, X must be a superkey
  • Eliminates all non-trivial functional dependencies where determinant is not a superkey
-- Example: 3NF but not BCNF
-- Table: StudentAdvisor(student_id, advisor, advisor_dept)
-- FD1: student_id ‚Üí advisor
-- FD2: advisor ‚Üí advisor_dept
-- advisor is not a superkey but determines advisor_dept

Explain different types of joins in SQL with examples

‚ñº

SQL Joins combine rows from two or more tables based on related columns.

-- Sample Tables
Employees(emp_id, emp_name, dept_id)
Departments(dept_id, dept_name, location)

1. INNER JOIN: Returns matching rows from both tables

SELECT e.emp_name, d.dept_name
FROM Employees e
INNER JOIN Departments d ON e.dept_id = d.dept_id;

2. LEFT JOIN (LEFT OUTER JOIN): All rows from left table + matching rows from right

SELECT e.emp_name, d.dept_name
FROM Employees e
LEFT JOIN Departments d ON e.dept_id = d.dept_id;

3. RIGHT JOIN (RIGHT OUTER JOIN): All rows from right table + matching rows from left

SELECT e.emp_name, d.dept_name
FROM Employees e
RIGHT JOIN Departments d ON e.dept_id = d.dept_id;

4. FULL OUTER JOIN: All rows when there's a match in either table

SELECT e.emp_name, d.dept_name
FROM Employees e
FULL OUTER JOIN Departments d ON e.dept_id = d.dept_id;

5. CROSS JOIN: Cartesian product of both tables

SELECT e.emp_name, d.dept_name
FROM Employees e
CROSS JOIN Departments d;

6. SELF JOIN: Join a table with itself

SELECT e1.emp_name AS Employee, e2.emp_name AS Manager
FROM Employees e1
LEFT JOIN Employees e2 ON e1.manager_id = e2.emp_id;

What is indexing in databases and explain different types of indexes?

‚ñº

Indexing is a database optimization technique that speeds up data retrieval operations.

Types of Indexes:

1. Primary Index:

  • Created automatically with primary key constraint
  • Determines physical ordering of data in storage
  • Unique and not null

2. Secondary Index:

  • Created on non-primary key columns
  • Doesn't affect physical storage order
  • Can be unique or non-unique

3. Clustered Index:

  • Determines physical order of data in table
  • Only one clustered index per table
  • Faster for range queries
  • Example: Primary key index in most DBMS

4. Non-Clustered Index:

  • Separate structure from data storage
  • Contains pointers to actual data
  • Multiple non-clustered indexes per table
  • Example: Secondary indexes

5. B-Tree Index (Balanced Tree):

  • Most common index structure
  • Self-balancing tree structure
  • Efficient for equality and range searches
  • Used by most relational databases

6. Hash Index:

  • Uses hash function to map keys to locations
  • Very fast for equality searches (O(1))
  • Poor for range queries
  • Used in in-memory databases

7. Bitmap Index:

  • Uses bit arrays (bitmaps)
  • Efficient for low-cardinality columns
  • Good for data warehousing
  • Supports fast Boolean operations

8. Full-Text Index:

  • Designed for text search
  • Supports linguistic search, stemming, synonyms
  • Used in document retrieval systems
-- Creating different types of indexes
CREATE INDEX idx_email ON users(email); -- Secondary index
CREATE UNIQUE INDEX idx_username ON users(username); -- Unique index
CREATE INDEX idx_name_dob ON users(last_name, first_name, date_of_birth); -- Composite index
CREATE FULLTEXT INDEX idx_content ON articles(content); -- Full-text index
